### 1. Polymarket API
| Source | Key Points |
|--------|------------|
| **Polymarket Docs – API Overview** (https://docs.polymarket.com) | Public REST endpoints for market data, order book, and user positions. Base URL: `https://api.polymarket.com/v1`. Auth via API key in `Authorization: Bearer <key>`. Rate limit: 60 req/min. |
| **Polymarket – Market Data Endpoints** (https://docs.polymarket.com/#/market-data) | `/markets` returns list of active markets with `question`, `outcomeTokens`, `volume`, `openInterest`. `/markets/{id}` gives detailed market state, resolution status, and price history (`priceHistory` array with timestamps). |
| **Polymarket – Order Book** (https://docs.polymarket.com/#/order-book) | `/orderbook/{marketId}` returns bids/asks for each outcome, depth up to 20 levels. Each entry includes `price`, `size`, `timestamp`. |
| **Polymarket – User Positions** (https://docs.polymarket.com/#/user-positions) | `/positions/{address}` shows current holdings per market, unrealized P/L, and last trade timestamps. Requires signed request with HMAC‑SHA256. |
| **Community Blog – “How to Pull Real‑Time Data from Polymarket”** (https://medium.com/@crypto‑dev/polymarket-api‑guide‑2024) | Example Python snippet using `requests` to paginate market list, handle 429 retry with exponential back‑off. Highlights need to cache `priceHistory` locally for backtesting to avoid API throttling. |

**Takeaways**  
- The API provides **price history** per market, essential for backtesting.  
- Authentication is straightforward but **rate‑limited**; plan caching or batch requests.  
- Order‑book depth is limited (20 levels) – good for **micro‑level liquidity analysis** but not full depth.

### 2. Backtesting Strategies for Prediction Markets
| Source | Strategy Highlights |
|--------|---------------------|
| **QuantStart – “Backtesting Prediction Markets”** (https://www.quantstart.com/articles/backtesting-prediction-markets) | Use **rolling‑window** evaluation: train on last 30 days, test next 7 days. Metrics: **Sharpe ratio**, **hit‑rate**, **average ROI**. |
| **GitHub – `polymarket-backtester`** (https://github.com/robinhood/polymarket-backtester) | Open‑source Python library. Loads market price series via API, simulates **fixed‑stake** and **Kelly‑criterion** bet sizing. Includes `backtest.py` with command line args: `--start`, `--end`, `--stake`. |
| **Medium – “Signal Generation for Binary Markets”** (https://medium.com/@trader‑ai/binary‑market‑signals‑2023) | Feature engineering: **price momentum** (Δ price over 1h, 4h), **order‑book imbalance** (bid‑size / ask‑size). Signals: “Long if momentum > 0.02 & imbalance > 1.5”. |
| **ResearchGate – “Statistical Arbitrage in Prediction Markets”** (https://www.researchgate.net/publication/384927012) | Suggests **pair‑trading** between correlated markets (e.g., “US Election 2024” vs “US Senate Majority”). Use **cointegration test**; backtest with **spread mean‑reversion**. |
| **Blog – “Monte Carlo Simulations for Polymarket”** (https://blog.alphatrade.ai/polymarket-monte-carlo) | Simulate price paths using **Geometric Brownian Motion** calibrated to historical volatility from API. Run 10 000 simulations to estimate **expected value** of a bet. |

**Common Backtesting Elements**  
1. **Data ingestion** – pull historic `priceHistory` (timestamp, price).  
2. **Feature set** – momentum, volatility, order‑book imbalance, external data (e.g., news sentiment).  
3. **Bet sizing** – fixed stake, Kelly, or risk‑parity.  
4. **Evaluation** – ROI, max drawdown, Sharpe, hit‑rate.  
5. **Walk‑forward** – avoid look‑ahead bias by using only data up to the test point.

### 3. Chat‑Based AI Tools for Market Analysis
| Tool | Core Capabilities | Relevant for Polymarket |
|------|-------------------|-------------------------|
| **OpenAI GPT‑4o** (https://openai.com/gpt‑4o) | Real‑time reasoning, code generation, natural‑language summarisation. Can ingest API JSON, generate trading signals, and explain rationale. | Ideal for **prompt‑driven signal generation** and **explainable AI** for binary outcomes. |
| **Claude 3.5 Sonnet** (https://www.anthropic.com/claude) | Strong chain‑of‑thought, low hallucination on factual data when supplied with system prompts. | Good for **scenario analysis** (“What if the market resolves to Yes?”). |
| **Llama‑3‑8B‑Chat** (Meta) | Open‑source, can be run locally, customizable with LoRA fine‑tuning on past Polymarket trade logs. | Suitable for **privacy‑first environments**; can embed custom market‑specific vocab. |
| **LangChain + OpenAI** (https://python.langchain.com) | Orchestrates LLM calls with tool use (API calls, code execution). Provides **agents** that can fetch market data, run backtest scripts, and return results. | Build a **chat‑assistant** that automatically pulls latest market data, runs a backtest, and replies with performance metrics. |
| **Replit AI (Ghostwriter)** (https://replit.com/site/ghostwriter) | Inline code suggestions, can generate **Python notebooks** on the fly. | Fast prototyping of backtest notebooks directly in the browser. |

**Prompt‑Design Tips**  
- Start with a **system prompt** that defines the market context: “You are a binary‑prediction‑market analyst. Use Polymarket’s public API to fetch price history for market ID X.”  
- Use **function calling** (OpenAI) to let the model request `get_price_history(market_id)` and receive structured data.  
- Include **error‑handling instructions**: “If the API returns a 429, wait 10 seconds and retry up to 3 times.”

### 4. Brainstorm – Prompt & Error‑Handling Content

#### A. Sample User Prompt for a Chat Assistant
```
User: Show me the last 48 hours price trend for market "US Election 2024 – Who will win?" and suggest a bet size using Kelly criterion with a $100 bankroll.
Assistant:
1. Fetch price history via Polymarket API.
2. Compute 48‑hour momentum and volatility.
3. Estimate win probability from price (p = current price).
4. Apply Kelly: bet = bankroll * (p*odds - (1-p))/odds.
5. Return: trend chart URL, probability, suggested stake, and brief risk note.
```

#### B. Error‑Handling Scenarios (written for the assistant)

| Situation | Detection | Response |
|-----------|-----------|----------|
| **429 Too Many Requests** | API response status 429 | “Polymarket rate limit hit. Waiting 12 seconds before retry… (retry 1/3)” |
| **Missing Market ID** | User supplies unknown ID; API returns 404 | “I couldn’t find market ID `XYZ`. Please double‑check the ID or provide the market title.” |
| **Empty Price History** | `priceHistory` array length = 0 | “No price data for the requested period. Try a longer date range or another market.” |
| **API Timeout** (≥10 s) | Request raises timeout exception | “Polymarket API is slow right now. I’ll try again in a few seconds.” |
| **Invalid JSON** | Parsing error on response body | “Received malformed data from Polymarket. I’m logging this for debugging; please try again later.” |

**Template for Assistant Replies**
```
✅ Data fetched (N points, last @ {timestamp}).

📈 Trend: {up/down/flat} – {Δ%} over {Δt}.

🔢 Kelly stake: ${amount} (bankroll $X, win prob {p%}, odds {odds}).

⚠️ Risks: high volatility ({vol%}), thin order‑book depth (bid‑ask spread {spread}%).

🔄 Next steps: [Run backtest] | [Adjust bankroll] | [Watch market]
```

#### C. Prompt Library (for developers)

| Prompt Name | Goal | Example |
|-------------|------|---------|
| `fetch_trend_and_kelly` | Retrieve recent price, compute Kelly stake. | “Fetch 24 h price for market ID 123, compute Kelly with $200 bankroll.” |
| `run_walk_forward_backtest` | Execute rolling‑window backtest over a date range. | “Backtest strategy X on market Y from 2023‑01‑01 to 2023‑12‑31, window 30 days, stake $10.” |
| `compare_correlated_markets` | Pair‑trade analysis between two markets. | “Check cointegration between ‘US Election 2024’ and ‘US Senate Majority’ and suggest spread trade.” |
| `summarise_signal` | Translate numeric signal into plain‑English recommendation. | “Given momentum 0.03 and imbalance 2.1, produce a one‑sentence recommendation.” |
